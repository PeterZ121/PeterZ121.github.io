NewSQL和TiDB理论

发表于 2022-03-09 更新于 2022-03-17

数据库类型

商用关系型数据库：Oracle DB2 Sybase SQL Server

开源关系型数据库：MySQL PostgreSQL

大数据生态NoSQL(非结构化数据库Not Only SQL)：Hadoop Redis MangoDB

集成分布式技术与关系模型的NewSQL：Spanner TiDB 非结构化存储+支持事务

分布式系统

分布式系统：提高硬件量而非提高硬件性能，来满足数据爆发式增长

谷歌数据三驾马车：

  • GFS(Google File System)：解决分布式文件系统问题
  • Bigtable ：解决分布式Key-Value存储问题
  • MapReduce：解决在分布式文件系统、分布式KV存储上如何做分布式计算和分析的问题

RPO：数据恢复点目标，业务系统所能容忍的数据丢失量

RTO：恢复时间目标，业务系统停止服务的最长时间

HTAP：混合数据服务，海量数据下OLTP OLAP的数据服务融合

TiDB

特点

  • 计算与存储分离，存储引擎为TiKV

架构

  • 支持标准SQL的计算引擎 TiDB-Server(兼容MySQL 5.7)

  • 存储层：分布式存储引擎 TiKV

  • 负责元信息管理与调度的引擎 PD(Placement Driver)

    • 集群的元信息管理 （分片分布、拓扑结构等）
    • 分布式事务ID的分配
    • 调度中心

数据结构

  • LSM-tree结构：顺序写入，写入到一定阈值后，持久化成静态文件，随着写入的越来越大，静态文件被推到下一层进行再合并。其本质是用空间置换写入延迟的思想

  • RocksDB：一款成熟的LSM-tree存储引擎。支持批量写入，无锁快照读

  • TiKV节点选择基于LSM-tree的RocksDB引擎

选择数据副本

选择强一致的算法：Raft

如何实现扩展——做数据分片

自动分片

采用range算法

把TiKV看做一个巨大的有序的key-value的Map，通过范围的方式将Key-Value空间分成若干段，每一段是连续的Key-Value,称作Region

可通过查询找到某个key在哪个Region中，在哪个TiKV Node中（每个Region只能保存在一个物理节点中，其调度由PD负责）

TiKV整体架构

  • 每个分片通过Raft进行复制，默认是三个副本，其中一个副本是Raft leader
  • TiKV通过PD对这些Region及其副本进行调度，并决策哪个副本成为leader，以保证数据和读写负载都均匀地分散在不同的TiKV节点上
