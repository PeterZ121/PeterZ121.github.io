区块链网络

发表于 2021-11-15 更新于 2021-11-16

接下来会在概念层面上描述 Hyperledger Fabric 是如何让组织间以区块链网络的形式进行合作的。

什么是区块链网络

区块链网络是一个为应用程序提供账本和智能合约服务的技术基础设施。

首先，智能合约被用来生成交易，接下来这些交易会被分发给网络中的每个节点，这些交易会被记录在他们的账本副本上且不可篡改。
一般多个组织聚集在一起作为一个联盟来形成一个网络。

创建网络

让我们从头开始创建网络的基础吧！

图中有四个元素：

  • 排序服务O4
  • 网络配置NC4
  • 组织R4
  • 证书颁发机构CA4

往下
当一个排序服务启动后就形成了如图的网络N。其中排序服务o4由一个单独的节点组成，是根据网络配置NC4来进行配置的。证书颁发机构CA4被用来向管理员和组织R4的网络节点分配身份信息（具体后面会讲到）

可以看到定义网络N的时候，第一件事就是定义一个排序服务。O4 最初被配置并且由组织 R4 的一个管理员来启动，并且由 R4 管理，配置 NC4 包含了描述网络管理能力初始集合的规则。最初在网络中集合仅赋予了 R4 这个权利。
目前 R4 是这个网络中唯一的一个成员

添加网络管理员

目前 NC4 最初被配置为仅仅允许 R4 用户在网络中具有管理的权限,接下来，允许R1用户也具有管理的权限。

新增元素： 组织R1和它的证书颁发机构CA1

组织 R4 更新了网络配置来使组织 R1 也成为了管理员。现在 R1 和 R4 在网络配置中便具有了相同的权限。
证书颁发机构 CA1 也被添加进来了，他用来标识 R1 组织的用户。现在从 R1 和 R4 来的用户就已经是网络的管理员了。

定义联盟

新增元素：联盟X1和其中的联盟成员——组织R1,R2、对应CA

  • 网络管理员定义了一个包含两个成员的联盟 X1，包含组织R1 和 R2 ,CA1 和 CA2 是这两个组织对应的证书颁发机构。
  • 这个联盟的定义被存储在了网络配置 NC4 中，会在接下来的网络开发中被使用。
    为什么联盟这么重要？我们能够看到联盟定义了网络中的一部分组织，他们共享了彼此能够交易的需求，在这个示例中就是 R1 和 R2 能够进行交易。


为联盟创建通道

通道——联盟中的成员彼此进行通信的主要机制。在一个网络中可能会有多个通道。

新增元素：通道C1，通道配置CC1

第一个通道是如何被添加到这个网络中的呢？

  • 通道C1——联盟X1为R1和R2创建
  • 通道配置CC1——控制管理通道C1，且完全独立于网络配置
  • CC1 是由 R1 和 R2 管理的，他们在 C1 上具有同等的权利。R4 在 CC1 中是没有任何权利的。因为他不是联盟成员
  • 该通道仅能处理在 R1 和 R2 之间进行的交易
  • 一个通道可以包含任意数量的组织，目前只包含了两个组织R1,R2

通道的重要性

  • 提供了一个联盟成员之间进行私有通信和私有数据的机制。实现了Fabric允许组织间共享基础设施的同时又保持了私有性。好比QQ应用（区块链网络）中的一个群聊（联盟），不同群聊间信息隐蔽不可见。
  • 通道实现了“从网络中解放出来”——网络配置 NC4 的任何改动都不会对通道配置 CC1 造成任何直接的影响，只有通道配置中指定的组织(QQ群聊认定的管理员)才能控制它


节点和账本

接下来使用通道将区块链网络以及组织的组件关联到一起

新增元素：节点P1，账本L1

  • Peer节点P1加入了通道C1。
  • P1存储账本L1的副本
  • P1和O4使用通道C1进行通信

可以发现P1与CA1同为紫色，表明是CA1为P1颁发的身份信息，他将P1和组织R1关联起来。P1启动后，它就向排序服务O4发出请求加入通道C1。O4收到加入请求，就会使用通道配置CC1来决定P1在这个通道的权限，比如：CC1 决定 P1 是否能够向账本 L1 中读取或写入信息。

应用程序和智能合约链码

现在通道C1拥有了一个账本，我们可以连接客户端应用来使用由 Peer 节点提供的服务了。

新增元素：智能合约S5，客户端应用A1

  • 智能合约S5被安装在P1
  • 在组织R1中的客户端应用A1可以通过P1使用S5来访问账本
  • A1、P1、O4都加入了通道C1，他们都可以使用这个通道提供的通信设备
  • 尽管客户端应用处在Fabric区块链网络的外边，但它是可以通过通道C1跟网络相连的

就像Peer节点和排序节点一样，客户端应用也会有一个是它和组织相关联的身份信息

安装链码包

智能合约会被打包进一个链码中，这个链码会被部署到一个区块链网络中。可以把智能合约想象为管理交易，链码则管理着智能合约应该如何被打包部署。
在智能合约S5被开发完后，组织 R1 中的管理员必须要把它安装到节点 P1 上，当完成之后，P1 就完全了解了 S5。
当一个组织在一个通道中有多个 Peer 节点时，可以选择在哪个节点安装智能合约，而不需要每个 Peer 节点上都安装。

定义链码

  • 链码被安装在Peer节点上
  • 链码在一个通道范围内被管理和维护
  • 一个组织必须要批准一个链码定义才能使用已经安装的智能合约来查询账本和为交易
    因此在这里，管理员R1必须要批准S5的链码定义（即将链码定义提交到通道上），提交后，S5就可以被客户端应用A1调用了。
  • 智能合约具有封装性：通道C1上每个组件都可以访问S5，但不能访问其程序

背书策略

  • 背书策略在链码定义提供的信息中
  • 描述了在交易被其他组织存储在账本副本之前，哪些组织必须要同意此交易
    此处只有R1和R2对交易进行背书后，交易才能被存储到账本L1中
  • 链码定义提交到通道的同时，背书策略也被放置在通道账本上，，，，通道中的每个成员都可访问该策略

调用智能合约

智能合约能够被客户端应用调用的条件：

  • 被安装在Peer节点上

  • 在通道上被定义（即将链码定义提交到通道上）（即批准链码定义）

      问：链码定义是怎么被批准的？
  答：在联盟中，有效数量的组织都将链码定义提交到通道C1上

完成网络

网络开发的下一个阶段是将组织R2的基础设备添加到网络中

新增元素：P2节点、账本L1的一个副本、智能合约S5、客户端应用A2

  • R2添加了Peer节点P2
  • P2存有账本L1的一个副本，和链码S5
  • R2像R1一样批准了相同的链码定义
  • A1 和 A2 能够使用 Peer 节点 P1 或者 P2 来调用在 C1 上的 S5
    现在A1和A2就能够使用在通道C1上的智能合约S5和账本L1来生成交易了（组织R1和R2进行交易）

生成并接受交易

  • 节点只有安装了智能合约才能够运行
  • 在一个大网络中，不是所有Peer节点都有存储智能合约的副本，因此可以通过连接到通道来获取一个智能合约的接口信息
  • 所有Peer节点都可以验证并接受或者拒绝交易存入他们的账本 L1 的副本中，但只有安装了智能合约的Peer节点才能够参与交易背书的流程。这是生成一笔有效交易的核心。

Peer节点的类型

  • 提交节点——通道中的每一个Peer节点都是提交节点。

    作用：接受生成的区块，并在区块被验证之后附加到该节点的账本副本中

  • 背书节点——每个安装了智能合约并且该智能合约被客户端应用使用，来生成一个被签名的交易响应的Peer节点都是一个背书节点
    智能合约的背书策略明确了在交易被接受并且记录到提交节点的账本之前，需要哪些组织的 Peer 节点为交易签名。

  • 主节点与锚节点（待完成）

向通道中添加组合和节点

  • 通道中新的组织批准了通道中其他成员已经同意的链码参数之后就可以使用链码了。
  • 如果 R2 想改变链码的定义，那么 R1 和 R2 需要为他们的组织批准一个新的定义，然后其中的一个组织需要将定义提交到通道。
  • 网络及通道的策略在描绘去中心化网络中的自主和管控之间创建了平衡。

添加另外一个Peer节点

客户端应用程序A3能够使用C2,同节点P3和排序服务O4进行通信

  • L1和L2是完全相互独立的，被分别限制在各自的通道内
  • 目前，我们有一个独立的网络其中定义了两个完全。这些通道为组织提供了独立的管理设施来彼此交易。体现了去中心化的特点


把一个Peer节点添加到多个通道中

  • 客户端应用程序 A1 能够使用通道 C1 与节点 P1 和 P2 以及排序服务 O4 进行通信。
  • 客户端应用程序 A2 可以使用通道 C1 与节点 P1 和 P2 进行通信，以及使用通道 C2 与节点 P2 和 P3 以及排序服务 O4 进行通信。
  • 客户端应用程序 A3 能够使用通道 C2 与节点 P3 和 P2 和排序服务 O4 进行通信。
  • 排序服务 O4 能够使用通道 C1 和 C2 的通信服务。
  • 通道配置 CC1 应用在了通道 C1 中，CC2 应用在了通道 C2 中

节点P2同时是两个通道的成员，并且通过不同的智能合约来处理不同的账本

需要注意的是，在通道配置CC1中包含的策略决定了P2在通道C1中进行交易的操作，通道配置CC2中的策略约束了P2在通道C2中的控制


排序服务

通过前面的学习，我们发现排序服务O4在整个网络创建中发挥着十分强的中心作用，它最初被用来创建这个网络，然后连接到网络中的每个通道。
这样似乎违背了区块链去中心化的初衷
不过，排序服务本身是可以完全去中心化的！
让我们看一个更加真实的排序服务节点配置：

  • 含有排序服务O1和O4
  • O1由R1提供，O4由R4提供
  • 网络配置NC4中定义了来自R1和R4的操作者的网络资源权限

可以看到这个排序服务和 Hyperledger Fabric 中的其他组件一样，是完全去中心化的，它们分别在组织 R1 和 R4 中运行。

去中心化的交易分发

排序服务还提供了另外一个关键的设施——交易的分发点
排序服务从应用程序中搜集背书过的交易，然后把这些交易进行排序并放进区块中，这些区块被分发到通道中的每一个Peer节点。在每个这样的节点中，交易不管是有效还是无效，都会被记录下来，并且本地账本副本也会更新。
注意此处排序服务节点O4扮演者两个不同角色

  1. 通道C1中——O4搜集交易（背书过）、排序交易、在通道中分发区块。
  2. 在网络N中——提供对网络资源的管理


修改策略(mod_policy)

一个策略的变化是有策略中的策略来管理的，即修改策略(mod_policy)。

我们此前以及使用了mod_policy来管理网络中的变化。例如：

创建网络初始时， 这时候，只有组织R4被允许管理网络（这是通过在网络配置NC4中把R4定义为唯一一个有权限来管理网络资源的组织来实现的）（R4在mod_policy中）。R4通过将R1添加到通道创建和联盟创建的策略中来实现允许R1来管理网络

R4甚至可以通过网络配置来给R1更大的权限！——例如将R1直接添加到mod_policy,这样R1就同样可以管理这个网络中的变更了,甚至移除R4的管理权限！！

需要注意：mod_policy的所有演进，都需要所有被引入组织的统一


网络已经完全形成了

我们使用更加紧凑的视觉语法来稍微重新组织一下这个网络，因为它能够更好地适应更大的一个拓扑结构：

  • Fabric区块链网络包括了两个应用程序通道以及一个排序通道
  • R1 R4负责排序通道
  • R1 R2负责蓝色的应用程序通道
  • R2 R3负责红色的应用程序通道
  • 客户端应用程序 A1 是组织 R1 的元素，CA1 是它的证书颁发机构
  • 组织 R2 的节点 P2 可以使用蓝色的通信设施，也可以使用红色的应用程序通道。
  • 每个应用程序通道具有它自己的通道配置，这里是 CC1 和 CC2。
  • 系统通道的通道配置是网络配置 NC4 的一部分。

我们已经在概念上构建了一个 Hyperledger Fabric 区块链网络实例的最后一部分了。我们创建了一个有四个组织的网络，带有两个通道和三个 Peer 节点，两个智能合约和一个排序服务。并由四个证书颁发机构来支撑。它为三个客户端应用程序提供了账本及智能合约服务，这些应用程序可以通过两个通道与账本和智能合约进行交互。
